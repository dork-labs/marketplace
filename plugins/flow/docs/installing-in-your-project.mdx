# Install /flow in your project

`/flow` is **portable**: one identifiable, installable unit any project can adopt. It is a
self-contained DorkOS Marketplace `plugin`-type package whose canonical home is this `plugins/flow/`
directory (there is no separate engine package). Because the DorkOS registry is a strict superset of
Claude Code's format, the same package installs through either front door. DorkOS itself is a
consumer that dogfoods the plugin.

<Callout type="info">
  **Status.** The plugin is self-contained and loads today via
  `claude --plugin-dir <path>/plugins/flow` (the interim dogfood path). The
  one-command install through the DorkOS Marketplace and Claude Code front doors
  below is the remaining "blessed install" work, tracked under **DOR-172** (spec
  #266 extracted the engine into this plugin). Treat the `dorkos install` /
  `claude plugin install` steps as the intended adopter experience, not a claim
  that every front door is wired today.
</Callout>

## What you install

The flow plugin is one package that contributes a coherent set of layers:

- **The stage commands** - `/flow` (orchestrator) plus `/flow:capture … /flow:done`, the one spine
  from a raw thought to shipped.
- **The generic stage skills** - eight tracker-agnostic skills that speak only the `WorkItem` model
  and its verbs. They carry no tracker strings; the only tracker-aware unit is the adapter you
  generate (see [Build your tracker adapter](/docs/guides/flow/building-your-adapter)).
- **The deterministic scripts** - the decision oracles (ranking, gating, calibration, recovery,
  dedup) ship as dependency-free runnable TypeScript under `scripts/` (executed directly via `node`,
  no build step). The stage skills _call_ them rather than re-deriving the ladders in prose, so the
  logic runs identically every time at ~0 tokens.
- **The schedulable tick** - a `flow-drain` skill that carries one item to its review gate, shipped
  `enabled: false`. You wire a scheduler to it when you want autonomy (see
  [Bring your own scheduler](/docs/guides/flow/bring-your-own-scheduler)).
- **`/flow:init`** - the first-run setup that turns a fresh install into a configured, tracker-wired
  workflow.

## Installing the package

You install the flow plugin the same way you install any DorkOS package: from the DorkOS Marketplace,
or through Claude Code's plugin command, since the package validates against both formats.

<Tabs items={['DorkOS Marketplace', 'Claude Code']}>

<Tab value="DorkOS Marketplace">

<Steps>

<Step>

### Install from the marketplace

```bash
dorkos install flow
```

DorkOS fetches the package, verifies its manifest, shows you the layers and permissions it
contributes, and asks for confirmation before writing any files. You can also install it from inside
DorkOS via Dork Hub. Read [the Marketplace guide](/docs/marketplace) for the full install model.

</Step>

</Steps>

</Tab>

<Tab value="Claude Code">

<Steps>

<Step>

### Install as a Claude Code plugin

```bash
claude plugin install flow
```

The package carries a `.claude-plugin/plugin.json`, so Claude Code installs it as a native plugin -
the commands, skills, and hooks register the same way they do under DorkOS. The DorkOS-specific
extras (the marketplace surface, the optional server seat) layer on when you run under DorkOS.

</Step>

</Steps>

</Tab>

</Tabs>

## First run routes through `/flow:init`

The plugin is inert until it knows your tracker and identity. You do not configure it by hand: the
**first time you invoke `/flow`, the orchestrator detects that config is missing and routes you to
`/flow:init`.** This first-run setup is designed to:

<Steps>
  <Step>
    **Scaffold config from the template.** It seeds a working `config.json` (the
    dials that govern the loop) from the committed `config.example.json`
    template, and adds a gitignored `config.local.json` for tracker tokens and
    any account binding. The plugin never ships a resolved `config.json` or
    `*.local.json` (both are gitignored), so neither your config nor your
    secrets can leak back into the plugin when you dogfood it via
    `--plugin-dir`.
  </Step>
  <Step>
    **Resolve your tracker, identity, and project** through the calibration
    ladder - which tracker (Linear via MCP, Linear via Composio, Jira, GitHub
    Issues, or other), which connection, and how work maps to your project.
  </Step>
  <Step>
    **Generate and validate your adapter.** It generates a concrete tracker
    adapter from the contract plus the closest reference adapter, then runs the
    conformance test until it passes. The generic engine never speaks to your
    tracker directly; it trusts this one validated adapter. See [Build your
    tracker adapter](/docs/guides/flow/building-your-adapter).
  </Step>
</Steps>

<Callout type="info">
  After `/flow:init` completes, the rest of `/flow` behaves exactly as it does
  in-repo: drive it by hand one stage at a time, drain the queue from your
  terminal with `/flow auto`, or schedule the tick for unattended progress. None
  of those paths change because you installed the plugin.
</Callout>

## Where things live after install

| Path                                  | What it is                                                                  | Committed?                 |
| ------------------------------------- | --------------------------------------------------------------------------- | -------------------------- |
| `config/config.example.json`          | The committed config template (`/flow:init` seeds from it).                 | Yes                        |
| `config/config.json`                  | Your resolved dials.                                                        | No (gitignored; generated) |
| `config/config.local.json`            | Tracker tokens and account binding.                                         | No (gitignored)            |
| `scripts/*.ts`                        | The shipped deterministic oracles + `validate-adapter.ts` (run via `node`). | Yes (from the plugin)      |
| The generated adapter                 | Your tracker-aware code, conforming to the contract; you own it.            | Yes                        |
| `flow-drain` (schedulable tick skill) | The autonomy tick, `enabled: false` until you wire a scheduler to it.       | Yes                        |
