# Bring your own scheduler

Once you have installed `/flow` and your adapter is green, you can let the loop carry work to your
review gate on its own. Autonomy is designed as **bring-your-own-scheduler**: the plugin ships a
single schedulable tick, and _something external fires it_. That something is either the DorkOS server
(the premium host) or a plain OS-cron / CI entry (server-free). The same tick definition serves both;
the scheduler is swappable.

<Callout type="info">
  **This page is the adopter's autonomy guide.** For the full taxonomy of ways to drive the loop
  inside DorkOS - the attended `/flow auto` terminal drain, the scheduled Pulse seat, and the
  portable watcher - and for the invariants that hold across all of them (the always-on human-review
  gate and the non-trimmable calibration floor), read [Turning on the automatic
  loop](/docs/guides/flow/turning-on-autonomy). This page focuses on the one decision an adopter
  faces: *what fires the tick.*
</Callout>

## The tick is a schedulable skill, off by default

The autonomy tick ships as a `flow-drain` skill carrying `cron` and `enabled: false` frontmatter. Its
body delegates to the canonical single tick (`runTick`): walk the reconcilers in priority order
(`recovery → inbox/resume → dispatch`), claim one ready item, carry it to its review gate, and stop.
One tick equals one item. It is **not** `/flow auto` (which loops via the drain sentinel and the Stop
hook); the scheduler - not the skill - provides the repetition.

<Callout type="warn">
  **The plugin defines the tick but cannot fire itself.** Claude Code has no native scheduler
  (plugin "monitors" are session-scoped event streams, not crons), so a portable plugin cannot
  self-schedule. That is by design: the tick ships `enabled: false`, and turning on autonomy is the
  explicit act of wiring a scheduler to it. Nothing runs in the background until you do.
</Callout>

## Two ways to fire it

<Tabs items={['DorkOS server (premium host)', 'OS-cron / CI (server-free)']}>

<Tab value="DorkOS server (premium host)">

The DorkOS server is the **premium host** for the tick. It is not a prerequisite for autonomy - it is
the richest place to run it. Under the server you get a real UI, a token-efficient idle pre-check, and
the production-grade safety guards.

<Steps>
  <Step>
    **Run the DorkOS server and register your project's agent.** The server hosts the watcher and
    the file-based scheduler that fires `flow-drain`.
  </Step>
  <Step>
    **Enable the tick.** It ships `enabled: false`; turn it on. There is no build step and no
    migration - enabling the file is picked up live.
  </Step>
  <Step>
    **Tune cadence and limits** in the `loops` and `autonomy` config blocks. See [The
    dials](/docs/guides/flow/the-dials).
  </Step>
  <Step>
    **Watch and steer** with `/flow:status`, and halt or lift every mode at once with `/flow pause`
    / `/flow resume`.
  </Step>
</Steps>

**What the premium host adds**

- **An idle pre-check.** Before spawning an agent, the server runs the deterministic dispatch oracle
  (`node scripts/dispatch.mjs`) on the queue. An empty or starved queue costs no agent boot - only a
  ready item spends tokens.
- **A UI surface.** `/flow:status` renders every in-flight item, every parked question, and the
  rationale behind each decision; pause and resume are one control.
- **Production-grade guards.** A production gate and a leader lock keep a scheduled tick from firing in
  the wrong environment or from running twice concurrently.

</Tab>

<Tab value="OS-cron / CI (server-free)">

You do not need the DorkOS server to run autonomously. Any external scheduler that can invoke the tick
works: a `cron` entry, a CI loop, or a small daemon. This is the portable path for a non-DorkOS repo.

<Steps>
  <Step>
    **Pick a scheduler you operate** - an OS `cron` line, a scheduled CI job, or a tiny poller. It
    is the only thing providing repetition.
  </Step>
  <Step>
    **Fire the tick each interval.** Each fire runs one `flow-drain` tick that claims the top-ranked
    ready item, carries it to its review gate, and exits. One tick, one item, a fresh session each
    time.
  </Step>
  <Step>
    **Park headless.** With no human at a terminal, a stop-and-ask parks on the tracker and nudges
    via the `comment-and-nudge` channel; your reply is detected on the next tick's inbox poll and
    the run resumes where it left off.
  </Step>
</Steps>

<Callout type="info">
  Server-free autonomy is opt-in and self-operated: you own the scheduler, so you also own its
  cadence and its uptime. Cloud 24/7 progress with your laptop closed needs a hosted runner (a CI
  loop or a deployed DorkOS server) - that is a property of where the scheduler runs, not of
  `/flow`.
</Callout>

</Tab>

</Tabs>

## Which should I wire?

| If you…                                                      | Fire the tick from…                  |
| ------------------------------------------------------------ | ------------------------------------ |
| Run DorkOS and want a UI, idle pre-checks, and safety guards | **The DorkOS server** (enable it)    |
| Want autonomy on a non-DorkOS repo with no server            | **OS-cron / CI**                     |
| Want to watch the queue drain live, attended                 | Neither - use `/flow auto` (no tick) |

Both scheduled paths carry work to the _same_ review gate and stop there; the only difference is who
keeps the loop ticking and how much surface wraps it. The bundled tick, the `enabled: false` default,
and the explicit scheduler wiring keep autonomy opt-in by construction.
